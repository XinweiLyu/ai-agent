# CloudBase Run 部署错误修复指南

## 问题概述

**错误码**: `SERVICE_VERSION_NOT_FOUND`  
**错误信息**: Service version not found  
**问题**: 部署本地代码到腾讯云托管时，版本创建失败

## 问题根本原因分析

### 1. 服务状态分析

通过查询云托管服务详情，发现：
- ✅ 服务已创建（`ai-agent-backend-v3`）
- ❌ **服务没有在线版本**（`OnlineVersionInfos: []`）

### 2. 核心问题

**问题 1: 端口配置不符合 CloudBase Run 规范**

**当前代码做了什么**：
```yaml
server:
  port: 8123  # 硬编码端口
```

**它应该做什么**：
```yaml
server:
  port: ${PORT:8123}  # 读取 PORT 环境变量，默认值 8123
```

**为什么会出现问题**：
- CloudBase Run 在部署时会动态注入 `PORT` 环境变量
- 虽然当前配置的端口是 8123，但平台可能会在运行时使用不同的端口
- 应用必须监听 `PORT` 环境变量指定的端口，而不是硬编码的端口
- 如果应用监听错误的端口，健康检查会失败，导致版本创建失败

**问题 2: Dockerfile 未优化**

**当前 Dockerfile 的问题**：
- 使用完整的 Maven 镜像作为运行镜像（体积大，约 800MB+）
- 构建和运行在同一阶段，无法利用 Docker 缓存优化
- 启动时间慢，资源消耗大

## 解决方案

### 修复 1: 端口配置修复

**文件**: `src/main/resources/application-prod.yml`

**修改前**:
```yaml
server:
  port: 8123
```

**修改后**:
```yaml
server:
  port: ${PORT:8123}  # 读取 PORT 环境变量，默认值 8123
```

**解释**：
- `${PORT:8123}` 是 Spring Boot 的环境变量语法
- 如果存在 `PORT` 环境变量，使用该值
- 如果不存在，使用默认值 `8123`
- 这样既支持本地开发（无 PORT 环境变量），也支持 CloudBase Run 部署（有 PORT 环境变量）

### 修复 2: Dockerfile 优化

**文件**: `Dockerfile`

**优化策略**：使用多阶段构建（Multi-stage Build）

**优势**：
1. **减小镜像体积**：运行镜像只包含 JRE，不包含 Maven（从 800MB+ 降至 200MB+）
2. **加快构建速度**：利用 Docker 缓存层，依赖下载和代码编译分离
3. **提高安全性**：运行镜像不包含构建工具，减少攻击面

**多阶段构建原理**：
```
阶段1（builder）: 使用完整 Maven 镜像构建 JAR
  ↓
阶段2（runtime）: 使用轻量级 JRE 镜像运行 JAR
```

## 费曼学习法：理解 CloudBase Run 部署机制

### 1. 什么是 CloudBase Run？

**简单类比**：
想象 CloudBase Run 是一个**智能的容器托管服务**，就像：
- **传统方式**：你自己租房子、买家具、装修（自己管理服务器）
- **CloudBase Run**：你拎包入住，房东（平台）已经准备好一切，你只需要告诉房东你的需求

### 2. 版本（Version）是什么？

**简单理解**：
- **服务（Service）** = 你的应用（比如 "ai-agent-backend"）
- **版本（Version）** = 你的应用的一次部署（比如 "v1.0"、"v2.0"）
- **在线版本（Online Version）** = 当前正在运行的版本

**类比**：
- 服务 = 一个应用商店中的应用
- 版本 = 应用的某个版本号（1.0、2.0、3.0）
- 在线版本 = 用户当前下载使用的版本

**为什么需要版本？**
- 可以同时存在多个版本（用于灰度发布、回滚等）
- 只有在线版本才会接收流量
- 如果服务没有在线版本，就像应用商店中的应用没有可下载版本

### 3. PORT 环境变量的作用

**简单理解**：
CloudBase Run 就像一个**智能路由器**：
- 外部请求 → 路由器（平台）→ 你的应用
- 路由器需要知道你的应用监听哪个端口
- 平台会动态分配端口，并通过 `PORT` 环境变量告诉你

**为什么不能硬编码端口？**
- 想象你住在一个公寓楼里：
  - 硬编码端口 = 你固定住在 8123 号房间
  - 使用 PORT 环境变量 = 房东告诉你住哪个房间，你就住哪个房间
- 如果硬编码，但平台分配了其他端口，就像你去了错误的房间，找不到你的应用

### 4. 部署流程（简化版）

```
1. 上传代码 → 平台收到你的代码
2. 构建镜像 → 平台根据 Dockerfile 构建容器镜像
3. 创建版本 → 平台尝试创建新版本
4. 健康检查 → 平台检查应用是否正常启动
   ├─ 如果健康检查通过 → 版本创建成功 ✅
   └─ 如果健康检查失败 → 版本创建失败 ❌
5. 流量切换 → 如果版本创建成功，将流量切换到新版本
```

**你的问题出在第 4 步**：
- 应用启动后，监听的是硬编码的 8123 端口
- 但平台可能分配了其他端口（或期望应用监听 PORT 环境变量）
- 健康检查失败 → 版本创建失败 → `SERVICE_VERSION_NOT_FOUND`

### 5. Dockerfile 多阶段构建的原理

**简单类比**：
- **单阶段构建** = 在工厂里生产，也在工厂里销售（工厂很大，成本高）
- **多阶段构建** = 在工厂里生产，在商店里销售（商店很小，成本低）

**技术细节**：
```
阶段1（builder）:
  - 使用 maven:3.9-amazoncorretto-21（包含 Maven + JDK，约 800MB）
  - 下载依赖、编译代码、打包 JAR
  - 输出：ai-agent-0.0.1-SNAPSHOT.jar

阶段2（runtime）:
  - 使用 openjdk:21-slim（只包含 JRE，约 200MB）
  - 从阶段1复制 JAR 文件
  - 运行 JAR
  - 最终镜像：只包含运行所需的内容（约 200MB）
```

**为什么这样更好？**
- **体积小**：最终镜像不包含构建工具（Maven），只包含运行环境（JRE）
- **速度快**：利用 Docker 缓存，依赖下载和代码编译分离
- **安全**：运行镜像不包含构建工具，减少攻击面

```bash
# 应用应该监听 3000 端口，而不是 8123
```

### 3. 重新部署

使用 CloudBase MCP 工具重新部署：

```bash
# 部署到现有服务
manageCloudRun deploy \
  --serverName ai-agent-backend-v3 \
  --targetPath D:\2025-1\IdeaProjects\ai-agent \
  --serverConfig '{"Cpu":1,"Mem":2,"MinNum":1,"MaxNum":5,"Port":8123}'
```

**重要提示**：
- `MinNum: 1` - 保持至少 1 个实例运行，避免冷启动延迟
- `Port: 8123` - 与配置的默认端口一致
- 应用会通过 `${PORT:8123}` 读取平台注入的端口

## 常见问题

### Q1: 为什么本地开发不需要 PORT 环境变量？

**A**: 本地开发时，Spring Boot 会使用默认值 `8123`。只有部署到 CloudBase Run 时，平台才会注入 `PORT` 环境变量。

### Q2: 如果平台分配的端口不是 8123 怎么办？

**A**: 应用会通过 `${PORT:8123}` 自动读取平台分配的端口。你不需要关心具体端口号，平台会自动处理路由。

### Q3: 多阶段构建会影响本地开发吗？

**A**: 不会。Dockerfile 只影响容器构建，不影响本地开发。本地开发仍然使用 `mvn spring-boot:run` 或直接运行 JAR。

### Q4: 如何查看部署日志？

**A**: 使用 CloudBase MCP 工具：
```bash
# 查询服务详情
queryCloudRun detail --serverName ai-agent-backend-v3

# 查看函数日志（如果配置了日志）
getFunctionLogs --name <function-name>
```
